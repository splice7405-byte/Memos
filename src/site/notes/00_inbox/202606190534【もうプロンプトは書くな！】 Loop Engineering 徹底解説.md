- 00:09 <https://youtu.be/aiQJ98bYoYM?t=0m9s> イントロダクション：「プロンプトを打つな、ループを設計しろ」
    - エンジニア「Skippy」の「エージェントにプロンプトを打つのをやめろ、ループを設計しろ」というポストが24時間で500万インプレッションを記録。
    - Claude Codeの開発者である「オリス」も、プロンプトを打たずにルート（ループ）を走らせているだけだと明言。
    - 人間が手動で1つずつプロンプトを打つ限界を打破する「ループ・エンジニアリング」の本質を解説。

- 01:12 <https://youtu.be/aiQJ98bYoYM?t=1m12s> ループを設計するとは？（概念のシフト）
    - **人間がループの外に出る**：これまでは人間がチャット欄で実行・結果確認・次の指示を手動で回していたが、その「回す役割」をプログラムに渡す。
    - **AIモデルは部品（関数）になる**：AIはおしゃべりの相手ではなく、プログラムから呼び出される一関数となり、外側の段取り（条件分岐など）は人間がプログラムで書く。
    - **基本の4ステップ**：
        1. 実行する
        2. 結果を観察する
        3. 次にどうするか判断する
        4. 繰り返す（ゴールに達するまでループ）

- 02:29 <https://youtu.be/aiQJ98bYoYM?t=2m29s> なぜ今、ループ・エンジニアリングなのか？（4つの発展段階）
    - AIエンジニアリングは以下の4段階の階段を上ってきた。
        1. **プロンプト・エンジニアリング（Sonnet 3.5初期）**：優秀なエンジニアになりきって、などの「1回きりの問いかけ」を工夫。
        2. **コンテキスト・エンジニアリング（Opus 4時代）**：どの資料をどの順番で渡すか、文脈そのものを設計。
        3. **ハーネス・エンジニアリング**：エージェント1回分の実行環境（道具、完了条件）の装備を整える。
        4. **ループ・エンジニアリング（現在）**：外側で自装（自律走行）する仕組みの設計。
    - 例（エラーバグを直す場合）：プロンプトなら「エラーを貼って直して」、コンテキストなら「関連ファイルも渡す」、ハーネスなら「テストツールを持たせ緑になったら完了」、ループなら「バグ発生で勝手に起動しPR（プルリクエスト）作成まで自走」。
    - 昔の自走AI（AutoGPTなど）との違い：昔は無限ループでコストが爆発した。現代は「止め方」を最初から設計する。

- 05:06 <https://youtu.be/aiQJ98bYoYM?t=5m6s> ループを構成する「6つの部品」
    - 1. **トリガー（起点）**
        - 人間ではなくシステム（スケジュール、GitHub Actions、Webhookなど）に始めさせる。
    - 2. **隔離（ワークツリー）**
        - 複数エージェントが同じフォルダで書き換えると衝突（マージ競合）が起きるため、`git worktree` などで専用の独立した作業コピーを切り出す。
    - 3. **スキルズ（お作法の明文化）**
        - `skills.md` などのファイルに、プロジェクト固有のルール（命名規則、デプロイ前の必須テストなど）を記載。これがないとAIは毎回ゼロから手順を考えてしまう。
    - 4. **役割分担（作成とチェックの分離）**
        - 「コードを作るエージェント」と「チェックするエージェント」を分ける。チェックはルール通りか見るだけなので、軽い（安価な）モデルで十分（コスト削減）。
    - 5. **コネクター（外部連携）**
        - MCP（Model Context Protocol）などを利用し、Slack、監視ツール、課題管理ツールといった外の世界の現実の状態を読めるように繋ぐ。
    - 6. **メモリ（外部記憶）**
        - MarkdownやJSON、SQLiteなどで進捗や結果を書き残す。モデルはコンテキストを忘れるが、リポジトリ（外部記憶）は忘れないため、昨日の続きから動かせる。

- 08:10 <https://youtu.be/aiQJ98bYoYM?t=8m10s> ループが自走する1日の流れ（具体例）
    - 朝6時にトリガー（CIエラー検知など）で起動 ➔ メモリを読み込み昨日の続きを確認 ➔ 隔離されたワークツリーを作成 ➔ スキルズ（お作法）に従い重いモデルが修正 ➔ 軽いモデルがテストとルールを採定 ➔ 合格したらコネクター経由でPR作成・チケット更新 ➔ メモリに書き残して終了（人間が出社する頃にはレビュー待ちのPRが並んでいる状態）。

- 09:14 <https://youtu.be/aiQJ98bYoYM?t=9m14s> 1番難しい「停止条件」の設計
    - 「止め方を決めていないループは、ただの請求書である」：寝ている間に無限ループするとお札が燃え続ける。
    - **必須の3つの安全装置（疑似コードの条件）**：
        - `while (回数が上限未満 && まだ進捗・変化があり && 予算が残っている)`
        1. **回数の上限**：最大10回などで物理的に止める。
        2. **進捗チェック**：直近数回で出力や挙動が変わっていなければ、行き詰まっていると判断し終了。
        3. **予算（トークン量・ドル建て）の上限**：上限に達したら強制終了。
    - 実際にUber社などでも、社員1人あたり月1500ドルまでの上限を設けるなど、暴走を防ぐ止め方の設計が真剣に行われている。

- 10:45 <https://youtu.be/aiQJ98bYoYM?t=10m45s> ループの種類：開いたループ vs 閉じたループ
    - **開いたループ（探検型）**：未知の調査や思考錯誤に向いているが、どこまで行くか・コストがいくらかかるか読めない。
    - **閉じたループ（登山型）**：あらかじめ通る道を決め、各チェックポイントで検証しながら進む。本番環境で回すなら予測可能なこちらが強い。

- 11:19 <https://youtu.be/aiQJ98bYoYM?t=11m19s> 本物の中 la の落とし穴（注意点・賛否両論）
    - 肯定派「プロンプトを1打ちしている人は価値の9割を捨てている」、否定派「クローンジョブ（定期実行）に帽子を被せただけ（昔からあるものの言い換え）」。
    - **ループエンジニアリングの3つの落とし穴**：
        1. **検証責任は人間に残る**：自動で出たPRが正しいとは限らない。確かめる義務は人間にある。
        2. **意図せぬ負債（コードの肥大化）**：挙動を読まずにマージし続けると、誰も中身を理解していないコードが光の速さで積み上がり、バグ発生時に原因究明で詰む。
        3. **思考の明け渡し**：自動化のボタンを押すだけになると、人間の判断力が鈍る。
    - 結論：技術者であり続ける意志を持ったまま、ハンドルは握った状態で自動化せよ。

- 13:09 <https://youtu.be/aiQJ98bYoYM?t=13m09s> 今日から始める最初の3ステップ
    - いきなり全部揃えようとせず、小さく始める。
    - **ステップ1：手で繰り返している作業を1つ選ぶ**
        - おすすめは「テストが落ちる ➔ エラーを読む ➔ AIに直させる ➔ 再テストして緑なら完了」の往復。最初は書き込み（送信・削除）をさせず、読む・提案する（リードオンリー）だけで回すと安全。
    - **ステップ2：停止条件（回数上限・予算上限）を最初に入れる**
        - 請求書爆発を防ぐため、絶対に後回しにしない。
    - **ステップ3：差分を毎回ちゃんと読む**
        - コードの負債を溜めないため、人間の検証の手は手放さない。
    - プロンプトは死んだのではなく「土台」になった。その上にループという一段を組めるかどうかが今後の差になる。
